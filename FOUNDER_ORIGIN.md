# Founder Origin Story

## Build the Work. Or Work the Room.

StagePort x CueR came from a real founder-time conflict: the builder cannot be everywhere opportunity appears.

While building the core governance and movement logic behind StagePort, the live-event floor became the problem itself. A technical founder has to choose between staying inside the work or manually working the room where collaborators, sponsors, partners, and proof may appear.

That tension became the product thesis.

CueR was built to hold live room context long enough for a human to make a better decision. It reads synthetic floor state, attendee intent, pass tier, zone pressure, and event mode. It proposes useful next actions. It does not approve them. Human authority remains the boundary.

Approved actions become receipts. Receipts accumulate into keychains. The room becomes operational memory instead of evaporating into missed introductions, forgotten context, and unverified decisions.

## Why This Matters

Most event systems preserve logistics:

- who registered
- who checked in
- who messaged
- what session happened

StagePort x CueR preserves the higher-value layer:

- who should meet
- why the introduction matters
- who approved an access or connection action
- what changed afterward
- what can be verified later

The category is **Operational Memory for Live Systems**.

## Boundary

This repository is a scoped hackathon evaluation kit. It uses synthetic data and public-safe schemas. It does not expose the full StagePort platform, private client environments, production signing infrastructure, commercial scoring logic, or unreleased StudioOS methodology.

The public evaluation instance demonstrates the proof loop:

```text
Agents propose.
Humans approve.
Receipts prove what changed.
```

## Evaluation Lens

For judges and technical reviewers, the founder story is not separate from the architecture. It explains the use case: live rooms create value faster than people can manually capture it.

StagePort x CueR turns that constraint into a system:

```text
room state -> CueR proposal -> human approval -> signed receipt -> keychain memory
```

The founder can keep building. The room can still create value. The decision trail survives.
